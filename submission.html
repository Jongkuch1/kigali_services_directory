Part 1 — Implementation Reflection

Getting Firebase and Firestore properly set up was probably the most time-consuming part of this project. Below I've written up the main problems I ran into — what the error looked like, what was actually causing it, and how I got past it.

1.1 Connecting the App to Firebase

Challenge The first time I ran the app after adding Firebase it crashed on startup without even showing a screen. I'd added the packages but hadn't actually connected them to my Firebase project yet.
Error message seen: "No Firebase App '[DEFAULT]' has been created — call Firebase.initializeApp()"
Resolution I installed the FlutterFire CLI and ran flutterfire configure from the project root. It pulled in the correct google-services.json and GoogleService-Info.plist for each platform and generated firebase_options.dart. I also had to update main() to call WidgetsFlutterBinding.ensureInitialized() and Firebase.initializeApp() before runApp() — without that order Firebase initialises too late and the same crash happens. After those changes the app connected to Firebase cleanly every time.

1.2 Email Verification Not Enforced After Sign-In

Challenge After implementing the sign-up and sign-in flows, I discovered that users who had registered but never clicked their verification email were still able to log in and navigate directly to the full app. The verification requirement was not being enforced.
What was happening: Firebase signs a user in the moment their password matches — it doesn't check whether the email address has been confirmed. My code was only testing whether someone was logged in at all, not whether they'd actually verified their address. So anyone who registered and ignored the verification email could still get into the app.
Resolution Inside _onAuthStateChanged() I added a user.reload() call before checking emailVerified. That reload matters — the SDK caches the user object locally, and without it you can read a stale value and think the email hasn't been verified when it actually has. After the reload, if emailVerified is still false, the auth status stays unauthenticated and an emailNotVerified flag gets set. The screen then shows a banner letting the user resend the email or tap a button once they've clicked the link. I also tightened isAuthenticated to require both an authenticated status and a verified email — no single point of failure.

1.3 Database Updates Not Reflecting in the UI

Challenge After completing the first working version of the Directory screen, I noticed that when I added a new listing, it did not appear in the list until the app was fully restarted. Edits and deletions similarly had no effect on what was shown on screen until the app was closed and reopened.
Root cause: I was calling Firestore's .get() when the screen opened and storing that result. The problem is .get() is a one-time read — it grabs a snapshot and stops. Any changes to the database after that point were just invisible to the app.
Resolution I switched from .get() to .snapshots() in FirestoreService. Snapshots keep a live connection open and push an update whenever the data changes — from any user. ListingsProvider subscribes to that stream and calls notifyListeners() each time it fires, so all watching screens rebuild with fresh data automatically. After this change, adds, edits, and deletes all showed up in real time without needing to restart anything.

1.4 Missing Query Index for User Listings

Challenge The My Listings screen — which shows only the current user's own listings — crashed at runtime immediately after I implemented the user-specific Firestore query. The Directory screen worked fine, but My Listings consistently threw an exception and displayed nothing.
Error message seen: "[cloud_firestore/failed-precondition] The query requires an index. You can create it here: https://console.firebase.google.com/..."
Resolution Firestore needs a composite index when a query filters on one field and sorts on another. My query was filtering by createdBy and ordering by timestamp, which requires a pre-built index that didn't exist yet. The helpful thing is Firebase puts a direct link in the error message that opens the index creation page in the console with everything already filled in. I clicked Create Index and waited a couple of minutes for it to build. After that the My Listings screen worked exactly like the Directory screen.

1.5 Google Maps Rendering as a Blank Grey Screen

Challenge When I first opened the Map View on the Android emulator the whole map area was just a grey box. No tiles, no labels, no markers — even though the listing data was loading fine.
Logcat message seen: "Failed to load map. Error contacting Google servers. This is probably an authentication issue (but could be due to network errors)."
Resolution I'd generated an API key and enabled the Maps SDK, but placed the meta-data tag in the wrong spot in the Android manifest — inside the activity element instead of the application element. Moving it fixed Android. For iOS I added GMSServices.provideAPIKey() in AppDelegate.swift before the super.application call. After restarting the emulator the map loaded properly and the markers from Firestore appeared in the right positions.

Part 4 — Design Summary Document

4.1 Firestore Database Structure

The app uses two top-level Firestore collections — users and listings — with reviews stored as a subcollection under each listing. I deliberately kept the structure flat; it makes querying simple and avoids the extra complexity that comes with deeply nested collections.

Collection: users

Each document in users is created as soon as someone registers. I use the Firebase Auth UID as the document ID rather than an auto-generated one — that way I can find a user's profile directly from their auth identity. It also keeps the security rules simple: each user can only read and write their own document.

FieldTypePurpose
uidStringFirebase Auth UID — matches the document ID
nameStringDisplay name entered at registration
emailStringEmail address used for authentication
createdAtTimestampAccount registration date and time

Collection: listings

Listing documents get auto-generated IDs from Firestore. The key field is createdBy, which stores the UID of whoever created the listing. That one field does two jobs: it lets the app filter listings for the My Listings tab, and the security rules use it to make sure only the creator can edit or delete their own entries. Any other signed-in user can read all listings, which is what makes the shared directory work.

FieldTypePurpose
nameStringName of the place or service
categoryStringCafés · Hospitals · Police · Libraries · Parks · Restaurants · Tourist
addressStringStreet address in Kigali
contactStringPhone number or contact information
descriptionStringFree-text description of the place or service
latNumberGeographic latitude in decimal degrees — used by the map marker
lngNumberGeographic longitude in decimal degrees — used by the map marker
ratingNumberAverage user rating from 0.0 to 5.0
createdByStringUID of the authenticated user who created the listing
timestampTimestampCreation date and time — used to sort the directory newest first

Subcollection: listings / {listingId} / reviews

Reviews sit directly under each listing as a subcollection. It keeps the data where it belongs and means I can load reviews for a listing without re-reading the listing document itself.

FieldTypePurpose
authorStringDisplay name of the reviewer
textStringFull review text
ratingNumberReviewer's score from 0.0 to 5.0
timestampTimestampDate and time the review was submitted

4.2 How Listings Are Modeled

All listing data flows through a ListingModel class in lib/models/listing_model.dart. It has two conversion methods: fromFirestore() turns a raw snapshot into a typed Dart object with null-safe fallbacks for every field, and toFirestore() turns it back into a map for saving. Nothing outside this class ever deals with raw Firestore maps directly.

This means all screens, providers, and widgets work with proper typed objects rather than raw maps with string keys. If the Firestore schema changes, only ListingModel needs updating — not every screen that uses listing data.

The id field is nullable because Firestore assigns it at write time. fromFirestore() populates it from the document ID, and it's needed for any update or delete call.

4.3 State Management Architecture

I used the Provider package with ChangeNotifier throughout. The goal was to keep Firebase calls out of the screen files entirely, so I split everything into three layers with data flowing in one direction:

Firebase / Firestore   →   Service Layer   →   Provider Layer   →   UI Layer

Service Layer

AuthService handles all Firebase Auth calls — sign-up, sign-in, sign-out, verification, and user reload. FirestoreService covers all Firestore work — streams, CRUD, reviews, and bookmarks. Neither class imports Flutter widgets or Provider, so they stay clean and are straightforward to test on their own.

Provider Layer

Three ChangeNotifier providers manage all application state:

  • AuthProvider listens to the Firebase auth state stream, enforces email verification, manages all auth states (loading, authenticated, error), and exposes the signed-in user's profile from Firestore.
  • ListingsProvider holds two live stream subscriptions — one for all listings, one for the current user's. The filteredListings getter applies both the search query and selected category against the in-memory list, so filtering is instant with no extra Firestore reads. CRUD calls go through FirestoreService, with errors surfaced to the UI via notifyListeners().
  • BookmarksProvider keeps a live stream on the user's bookmarks subcollection and exposes a toggle method that calls FirestoreService.toggleBookmark().

UI Layer

Screens use Consumer<T> or context.watch<T>() to read state and context.read<T>() for write operations. No screen file imports firebase_auth, firebase_core, or cloud_firestore — the UI stays fully decoupled from the backend.

4.4 Design Trade-offs and Technical Challenges

Client-side Filtering vs. Firestore Query Predicates

Both the name search and category filter run against the in-memory list rather than sending new Firestore queries on each keystroke or chip tap. Filtering is instant and costs no extra reads. The downside is the full listing list stays in memory for the whole session — for an app at this scale that's completely fine, but if the dataset grew very large, switching to server-side query predicates would be the right call.

Permanent Real-time Streams vs. On-demand Fetches

The two stream subscriptions start when the main screen loads and stay open for the whole session. That's what keeps all four tabs snappy — the data is already live, so switching tabs doesn't trigger any loading state. The main thing to watch for is cancelling those subscriptions on sign-out, which happens in ListingsProvider.cancelSubscriptions() called from the AuthProvider sign-out flow.

IndexedStack for Navigation vs. Navigator Pushes

Instead of pushing and popping screens on each tab change, I used an IndexedStack that keeps all four screens alive and just shows or hides them. Tab switching is immediate — no rebuilds, no flicker. It uses a bit more memory than a Navigator-based approach, but the UX improvement is noticeable.

Flat Collection Structure vs. User-nested Listings

All listings live in one shared top-level collection rather than nested under each user document. The main reason is the public directory — reading all listings is just one ordered collection query. User-specific queries still work fine via where('createdBy', isEqualTo: uid). Nesting listings under users would have needed Collection Group queries for the directory view, which felt like unnecessary complexity for what this app needs.
